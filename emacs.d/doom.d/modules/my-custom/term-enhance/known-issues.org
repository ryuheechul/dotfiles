#+TITLE: term-enhance known issues

Tracked problems found while testing tramp + terminal integration
(2026-07-04). Referenced from [[file:integration.el][integration.el]] and [[file:vterm.el][vterm.el]].

** DONE ghostel: "Forbidden reentrant call of Tramp" on any tramp buffer
Calling ~(ghostel)~ (or any wrapper here that reaches it) from a buffer
whose ~default-directory~ is remote threw this error every time -
deterministic and host-independent (=/ssh:localhost:= and a real Linux
remote alike). Originally misattributed to ghostel's own spawn path and
worked around by forcing vterm for remote buffers in
~term-enhance/backend~.

Root-caused 2026-07-10 (fresh isolated daemon, lock-transition tracing on
tramp's per-connection "locked" property, plus a backtrace captured at the
*original* signal point via an advice on ~tramp-signal-hook-function~ -
~with-tramp-debug-message~ re-signals errors, so a plain
~debug-on-error~/~handler-bind~ backtrace collapses to
~tramp-maybe-open-connection~ and points at the wrong place):

- tramp's ~make-process~ handler makes the terminal buffer itself the
  connection buffer for the dedicated "ghostel" ssh channel, so the whole
  login handshake gets INSERTED into the ghostel buffer while tramp holds
  that channel's connection lock inside
  ~tramp-accept-process-output~/~tramp-process-actions~.
- doom's global flycheck (~flycheck-global-modes~ was ~t~ = everywhere,
  and ~ghostel-mode~ derives from no parent, so no special-mode escape
  hatch applied) had installed its ~after-change-functions~ hook in that
  buffer. Each handshake insert fired ~flycheck-handle-change~ ->
  ~flycheck-may-check-automatically~ -> ~file-exists-p~ on the remote
  ~default-directory~ -> a nested tramp op on the locked channel ->
  "Forbidden reentrant call of Tramp".
- this also explains every earlier dead end: it's not a timer (tramp
  suspends those), not connection-share, not shell integration - it fires
  on *any* config where a remote-file-probing ~after-change-functions~
  hook lives in the terminal buffer. A minimal batch emacs (tramp +
  ghostel only, no flycheck) spawns remote ghostel fine, which is how the
  config - not ghostel - was implicated.

Fix: [[file:integration.el][integration.el]] excludes ~ghostel-mode~/~vterm-mode~ from
~flycheck-global-modes~ (a mode hook can't do it - global minor modes
enable via ~after-change-major-mode-hook~, which runs after mode hooks).
The remote-forces-vterm override in ~term-enhance/backend~ is removed;
~term-enhance/backend-priority~ now applies to remote buffers too.
Verified end-to-end against =/ssh:localhost:=: spawn, interactive prompt
(starship + full zsh rc), command round-trip.

Possibly still worth an upstream note to ghostel (and arguably tramp):
any package whose ~after-change-functions~ hook touches remote files can
break any tramp-spawned terminal this way; tramp could bind
~inhibit-modification-hooks~ while writing handshake output into the
connection buffer.

** DONE ghostel over tramp: shell loads, then exits (code 1) right away
Follow-up to the reentrant fix above: with spawning fixed, the remote shell
rendered its full prompt (zshrc completed) and then died 1-2 seconds later -
sentinel reported "exited abnormally with code 1", silently, with a shim-armed
zsh ~EXIT~ trap never firing.

Diagnosis trail (isolated daemon, =/ssh:localhost:=): a
~process-send-string~/~process-send-eof~ sniffer proved Emacs sends nothing
after tramp's exec, so the exit was remote-side; ~zsh -f -i~ survived, so it
was shell config; an env-var bisect (~DISTROBOX_ENTER_PATH~ skip) narrowed it
to [[file:../../../../../zsh/zlogin][zsh/zlogin]].

Root cause: zlogin's ssh branch (session persistence) guarded tramp only via
=TERM != dumb= - true for tramp's own dumb shells, but a terminal spawned
*over* tramp gets a real TERM (=xterm-ghostty= via ghostel's on-remote
probe). So the remote interactive shell took the is-ssh branch and ~exec~'d
[[file:../../../../../bin/terminal/tmux-attach-or-new.sh][tmux-attach-or-new.sh]], which resolved =tmux= to the
[[file:../../../../../bin/path/emacs/tmux][bin/path/emacs mask]] (INSIDE_EMACS present), which ~exit 1~s under ghostel.
~exec~ had already replaced zsh, hence: prompt appears, then the whole
session exits 1 with no exit trap. This never bit vterm+tramp because those
sessions predated ghostel's remote TERM probe (vterm remotes kept
tramp-flavored TERM handling), and never bit locally because local shells
aren't ~is-ssh~.

Fix: [[file:../../../../../zsh/zlogin][zsh/zlogin]] now also skips the block when ~INSIDE_EMACS~ contains
=tramp= (tramp appends =,tramp:VER=). ~znap prompt~ was briefly suspected
(xtrace pointed there, but only because znap's instant-prompt redirects
stderr mid-flight, hiding the real trail) - it is innocent and untouched.
Verified: remote ghostel shell survives past startup and round-trips
commands with the full zshrc, znap prompt included.

** DONE vterm's tramp shell resolution works on Linux remotes, not macOS ones
Confirmed on 2026-07-04 with a controlled two-host test (see
[[file:*ghostel: "Forbidden reentrant call of Tramp" on any tramp buffer][the item above]] for the same hosts):
- a real remote Linux host (~getent~ present): vterm resolves the correct
  remote shell automatically and a session there is fully interactive
  (real zsh, real starship prompt, command output verified round-trip).
- =/ssh:localhost:= (this Mac, ~getent~ absent - confirmed via
  ~"/bin/sh: getent: command not found"~): stuck at a bare =sh-3.2$=
  prompt instead of zsh.

Root cause, confirmed by reading vterm.el's actual resolution order
(~vterm--get-shell~):
#+begin_src emacs-lisp
(or (vterm--tramp-get-shell method)   ; per-method entry in vterm-tramp-shells
    (vterm--tramp-get-shell t)        ; catch-all entry
    (with-connection-local-variables shell-file-name)  ; tramp's own default
    vterm-shell)                      ; last resort
#+end_src
Default ~vterm-tramp-shells~ is ~'(("ssh" login-shell) ("scp" login-shell)
("docker" "/bin/sh"))~ - no fallback configured for =ssh=. On a real Linux
remote, =login-shell='s ~getent passwd $LOGNAME~ (run *on that host* via
tramp) succeeds and resolves at the first step. On macOS-as-remote, getent
doesn't exist at all - deterministic and host-independent, not a
localhost coincidence - so both ~vterm--tramp-get-shell~ calls return nil
and resolution falls to step 3: tramp hardcodes ~shell-file-name~ to
=/bin/sh= via a connection-local profile
(~tramp-connection-local-default-shell-variables~ in tramp-integration.el)
for *any* remote connection. That's what lands you at =sh=.

[[file:vterm.el][vterm.el]]'s ~vterm/reset-shell-for-tramp~ (setting the explicit ~vterm-shell~
variable) is step 4 - *never reached* here, since step 3 always resolves
to something non-nil first. It was never the mechanism at play for this
bug; it only matters for local sessions (keeping ~vterm-shell~ in sync
across opens, its original March 2025 purpose).

Fix applied: configure the fallback slot ~vterm-tramp-shells~ already
supports, rather than touching ~vterm/reset-shell-for-tramp~:
#+begin_src emacs-lisp
(setq vterm-tramp-shells '(("ssh" login-shell "zsh")))
#+end_src
The fallback slot is spliced unquoted into vterm's =exec %s= (unlike
ghostel's, which shell-quotes it), so a bare =zsh= does a =$PATH= lookup
on the remote shell rather than needing a hardcoded path - tested and
confirmed working.
