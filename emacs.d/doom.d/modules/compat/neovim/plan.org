#+TITLE: Plan: closing the neovim -> emacs gap
#+STARTUP: overview

* Read this first                                                 :meta:

This is a reference menu, not a sprint backlog. Nothing below is meant to
be worked through proactively or systematically - pick up a single item
only when a real, felt need for it actually comes up in daily use. "Let's
knock out Phase 1" is explicitly not the intent, even though the phases
are ordered by effort and risk, roughly smallest first.

This snapshot was written against the neovim config as it stood on
[2026-07-03]. That config keeps evolving on its own, so every reference
below to a specific nvim file, line, or behavior is provisional - re-check
the actual current nvim source before acting on any item, since the thing
being described may have moved, changed, or been removed entirely since
this was written.

* Goal

Make doom emacs feel like [[file:../../../../../nvim/][the neovim config]] for the things that matter -
muscle memory and quality of life - not a full port. Anything without a
daily-use payoff stays in neovim land (see [[*Not porting (deliberately)][not porting]]).

* Phase 1: muscle memory gaps (small, high frequency)

Ported from [[file:../../../../../nvim/lua/plugins/keymaps.lua][keymaps.lua]] unless noted. Each is a few lines in morevil.

** DONE ~gn~ / ~go~ -> jumplist forward/back
nvim: ~<C-I>~ / ~<C-O>~. Both bound to better-jumper in morevil. It turned
out ~go~ already behaved: doom remaps xref-go-back onto
~better-jumper-jump-backward~, which piggybacks evil's jump list (plain
motions like gg/G//search are tracked) - the real gap was only ~gn~,
which was unbound. Now both directions are explicit.

** DONE ~gl~ / ~gr~ / ~gd~ -> definition, references
nvim: ~gd~/~gr~ = lsp definition/references, ~gl~ = ~<C-]>~ via tagfunc.
root cause of "gd does nothing in lua" (2026-07-03): .lua files open in
lua-mode (doom's :lang lua has no +lsp flag) while ext-lang/lua hooked
lsp! only on lua-ts-mode, which nothing routes files to - eglot simply
never attached. fixed by hooking lua-mode too (ext-lang/lua/config.el);
gd needed no binding change (+lookup/definition all along). gl and gr
bound in morevil, shadowing accepted knowingly: gl was evil-lion's
left-align operator (gL survives), gr was doom's +eval:region (gR and
SPC-level eval bindings survive). follow-up (2026-07-04): gl and gd are
NOT the same thing in nvim - gd (vim.lsp.buf.definition) offers a
selection when the server reports several entities (lua-ls: name +
body), which doom's consult-xref picker mirrors correctly, while gl
(C-]) has TAG-jump semantics: follow the "link" to the FIRST match
immediately - and its real heritage is :help, where C-] follows the
help-tag hyperlink under the cursor. so gl is bound (:nm, so doc-ish
motion-state buffers get it too) to +neovim/goto-link (compat/neovim):
a button/Info link when on one, otherwise the first xref definition
with no picker; gd keeps the picker.

** DONE ~gx~ -> open URL under cursor (bugfix, 2026-07-10)
Was bound but broken on macOS: [[file:../../my-custom/morevil/config.el][morevil/config.el]] called
~browse-url-xdg-open~, which shells out to the ~xdg-open~ binary
unconditionally (Linux-only; errored "Searching for program: No such
file or directory, xdg-open"). Rebound to plain ~browse-url~, which
dispatches via ~browse-url-browser-function~ and auto-detects
~system-type~ (macOS: ~open~, Linux: xdg-open if present, ...) -
cross-platform without any extra code.

** DONE ~Y~ -> yank to end of line
~(setq evil-want-Y-yank-to-eol t)~ in this module's [[file:./init.el][init.el]] - module
init.el files run at doom's early-init stage, before any package (evil
included) loads, which is what the ~evil-want-*~ family requires;
config.el is too late. (evil's default Y = whole line; nvim's = y$.)

** DONE ~J~ -> join lines
Evil's active normal-state binding already handles ordinary adjacent lines
(pinned by ~parity/J-joins-lines~).  The TOML-specific failure was Doom's
comment-aware join advice: Emacs 30's ~toml-ts-mode~ recognizes =#= comments
through tree-sitter but leaves ~comment-start~ nil, so the advice passed nil
to ~comment-search-backward~.  :ext-lang toml now supplies TOML's local
comment delimiters, restoring nvim's =# one= + =# two= -> =# one two=; pinned
by ~parity/J-joins-toml-comments~.

** DONE ~j~/~k~ move by visual line when no count
Pairs with the existing ~global-visual-line-mode~:
~(setq evil-respect-visual-line-mode t)~ in this module's [[file:./init.el][init.el]] -
same before-evil-loads caveat as ~Y~.

follow-up (2026-07-14), operator-motion regression: that setting installs a
visual-line-mode keymap in MOTION state binding j/k to the EXCLUSIVE
~evil-next-visual-line~/~evil-previous-visual-line~ (evil-integration.el), and
operator-pending inherits motion state - so ~dj~/~dk~/~yj~/~yk~ stopped being
linewise over two full lines and became a 1-line exclusive delete that
dropped the adjacent line (a real muscle-memory hazard: delete the wrong
line). nvim avoids this by remapping j->gj in normal mode only, never
operator-pending; mirrored by restoring the linewise motions in OPERATOR
state ([[file:./config.el][config.el]], ~evil-define-minor-mode-key~ on ~'operator~) so operators
span whole lines again while bare j/k keep visual-line nav. pinned by
~parity/operator-jk-full-linewise~.

** DONE ~,~ -> repeat last ex command
nvim: ~@:~. Bound to ~evil-ex-repeat~ in morevil. Shadows evil's reverse
f/t repeat (same trade nvim made); ~;~ still repeats forward.

** DONE ~+~ / ~-~ -> increment/decrement number at point
nvim: ~<C-A>~/~<C-X>~. ~evil-numbers/inc-at-pt~ / ~dec-at-pt~ in morevil;
the package was already present transitively but is now declared in
morevil's packages.el so doom sync cannot autoremove it.

** TODO ~SPC RET~ -> run macro q
nvim: ~@q~. Emacs: ~(evil-execute-macro 1 (evil-get-register ?q))~.

** TODO ~SPC j~ / ~SPC k~ -> previous/next page
nvim: ~Hzb~ / ~Lzt~ trick. Emacs equivalent of the same trick:
~evil-window-top~ + ~evil-scroll-line-to-bottom~ (and the mirror), or simply
~evil-scroll-page-up/down~ if the exact cursor placement doesn't matter in
practice.

** DONE ~SPC b t~ -> trim trailing whitespace on buffer
~delete-trailing-whitespace~, bound in morevil (the key was free).

** DONE ~SPC t l~ -> toggle listchars
~whitespace-mode~ toggle (the mode itself is set up in this module).
conflicted with doom's ~doom/toggle-line-numbers~ - resolved by swapping:
listchars takes ~t l~ (nvim muscle memory), line numbers move to the
previously free and more mnemonic ~SPC t n~.

** DONE ~SPC t b~ -> toggle dark/light
Turned out to already exist: ~SPC t b~ was bound to
~toggle-doom-theme-tone~ ([[file:../../my-custom/term-enhance/theme.el][term-enhance/theme.el]]) all along.

** TODO ~SPC w z~ / ~SPC w m~ zoom semantics
Reuse the window-configuration save/restore toggle technique from
term-enhance's ~mux/zoom~ as a general ~SPC w z~; nvim's ~SPC w m~ (maximize
editor + outer mux) can call the ~mux-zoom~ script when ~$TMUX~ is set, like
morevil's C-hjkl tmux handoff does.

** DONE verify ~z=~ (fix spelling picker)
Confirmed working out of the box (2026-07-04) - doom's
~:checkers spell +flyspell~ covers it.

** DONE ~\s~ -> suspend to shell
nvim: ~:suspend~ (leader s). Real suspend is TTY-only: ~suspend-frame~ =
SIGTSTP, back with ~fg~, and correct for ~emacsclient -nw~ too. A GUI
frame has no parent shell (launchd started it), so there ~\s~ keeps
toggling the full-window terminal - whose shell aliases ~fg~ to toggle
back, completing the suspend/fg round trip. Dispatched per keypress on
~display-graphic-p~ in morevil.

** DONE dashboard ~i~ -> open a fresh unnamed buffer
nvim's dashboard offers this as an official menu entry (quick scratch
note, usually never saved; it does NOT enter insert state - that is up
to the user, so neither do we). ported the same way: a real
~+dashboard-menu-sections~ entry in compat/neovim (not morevil: it is a
dashboard behavior, not just a keybinding), with doom's own
~+default/new-buffer~ as the action - no custom command needed. the ~i~
binding must live in the evil normal AUX keymap of
~+dashboard-mode-map~: the menu renderer looks up its displayed key hint
there, and the dashboard's own ~[remap evil-insert] -> ignore~ lives
there too, swallowing ~i~ before any remap in the raw keymap is
consulted (first attempt failed exactly that way). the dashboard also
refuses insert state entirely now (advice on ~evil-insert-state~; an
earlier bug left it stuck in insert because evil-buffer-new's
set-window-buffer does not change the current buffer).

* Phase 2: QoL behaviors

From [[file:../../../../../nvim/lua/boot/misc.lua][boot/misc.lua]].

** DONE emacsclient preserves its caller directory
A no-file client used to retain the daemon's stale buffer-local
~default-directory~, so Magit could open a project from another client.
Emacs already receives the caller directory for every emacsclient process;
at ~server-after-make-frame-hook~, my-custom/client-enhance persists its
~server-client-directory~ on a per-client clone of Doom's dashboard buffer.
The isolated TTY client workspace is now also tied to its frame
and reclaimed in ~delete-frame-functions~, preventing unused =#N= workspaces
from accumulating.  Both contracts are pinned by client-enhance ERT tests.

** DONE autoread
nvim: ~autoread~ (+ shortmess tweak in boot/misc.lua) silently reloads a
file that changed outside the editor (git checkout, formatter, ...)
instead of prompting. emacs: ~global-auto-revert-mode~ +
~global-auto-revert-non-file-buffers~, in this module (doom does not
enable it; measured off).

** DONE help buffers act like ~less~
d/u/f/b scroll in ~help-mode~ now (morevil, next to the older Info-mode
map); verified they were NOT less-like before (d was literally ~ignore~).
nvim's auto-maximize of help windows not ported - doom's popup rules
could via ~set-popup-rule!~ if ever missed.

** DONE spell check on by default
verified doom's :checkers spell already hooks flyspell in prose modes
(org, markdown); added ~flyspell-prog-mode~ on prog-mode for the nvim
global-spell feel, scoped like nvim's treesitter @spell captures:
comments and docstrings but NOT plain strings
(~font-lock-string-face~ dropped from ~flyspell-prog-text-faces~ -
identifiers/paths in strings are all false positives). making it
actually WORK in TUI took two real fixes (2026-07-04): eglot semantic
tokens turn the face property into a LIST that the stock prog-mode
predicate's memq cannot see - overridden in compat/neovim; and the
default faces hide behind a supports-wave display guard that renders as
NOTHING on conservative terminals - replaced via
face-OVERRIDE-spec (a theme beats a defface, nothing beats the override)
with unconditional squiggles that leave text color alone: yellow3 for
misspellings, cyan3 for duplicates, identical in GUI and TUI (those are
the exact RGB the terminal palette gives "yellow"/"cyan"; cyan over the
stock DarkOrange because orange blurs into the misspelling yellow).
doom's flyspell-lazy was wrongly convicted along the way (a confounded
experiment: disabling it overlapped with the predicate fix landing); a
controlled re-test with lazy re-enabled showed TUI flagging works fine
under it, just idle-batched (~1s) - so lazy STAYS enabled, trading a
little lag on a never-urgent signal for keystroke-path headroom.

** DONE save-place
nvim: the LastPlace autocmd in boot/misc.lua reopens every file at the
last cursor position (gitcommit buffers excluded). emacs verification
came back negative: ~save-place-mode~ was OFF in this doom
checkout (the assumption that :config default enables it did not hold) -
enabled in compat/neovim. saveplace's default ignore regexp already
excludes COMMIT_EDITMSG, matching nvim's gitcommit exclusion.

** DONE verify smartcase search
confirmed: ~evil-ex-search-case~ is ~smart~ (2026-07-04), which is exactly
nvim's search config - both ~ignorecase~ AND ~smartcase~ set
([[file:../../../../../nvim/lua/boot/misc.lua][boot/misc.lua]]:35-36), not plain case-insensitivity. re-verified
functionally on a "case sensitive?" report (2026-07-13): the ~/~ key is
~evil-ex-search-forward~ (evil-search module), and "smart" means a
lowercase pattern folds case (~/banana~ finds ~BANANA~) while any uppercase
letter forces a sensitive match (~/Banana~ finds only ~Banana~) - same feel
as nvim, which reads as "insensitive" in daily use only because lowercase
input is the common case. lower-vs-upper behavior pinned by ~parity/~ test.

** DONE yank highlight
~:ui ophints~ (enabled).

** DONE undofile
~:emacs undo~ (enabled). completion+undo interaction verified on an "undo
felt strange after selecting a completion" report (2026-07-13): with
~evil-want-fine-undo~ nil an insert session is a single undo step, so
accepting a corfu candidate then ~u~ reverts the whole session in one go and
~C-r~ restores it (matching nvim's insert-session grouping), and corfu's
preview is not committed on ESC (doom quits corfu on insert-state exit). no
defect found across plain/cycled/replace/yasnippet-snippet paths; only the
async ~corfu-auto~ timing a batch daemon can't drive is left unprobed.
invariants pinned by the ~parity/corfu-undo-*~ / ~parity/corfu-escape-*~
tests.

** DONE clipboard sync
Emacs default + ~:os (tty +osc)~ for OSC 52 (enabled). OSC 52 turned
out to be write-only in practice (terminals refuse clipboard reads), so
TTY-frame paste reads ~pbpaste~ via ~+neovim/frame-aware-paste~ in
[[file:./config.el][config.el]] - GUI frames keep the native path (2026-07-11).

** DONE cursorline
Doom hl-line. cursorcolumn intentionally skipped: emacs has no
first-class equivalent and it mattered mostly alongside ~virtualedit=all~,
which can't be fully ported anyway (see [[file:./config.el][config.el]]).

** DONE colorcolumn (long-line hint) (2026-07-15)
a subtle vertical rule at the line-width limit, drawn by each editor's display
engine. emacs ([[file:./config.el][config.el]]): ~display-fill-column-indicator-mode~ (prog/text/conf) in
the ~fill-column-indicator~ face (a muted grey already - no override needed).
nvim ([[file:../../../../../nvim/lua/boot/misc.lua][nvim/lua/boot/misc.lua]]): ~colorcolumn=+0~, an offset from ~textwidth~.

*One source of truth for the width lives in [[file:../../../../../.editorconfig][.editorconfig]]* (~max_line_length =
100~, symlinked to ~~/.editorconfig~), the shared home for cross-editor
settings. neither editor hardcodes 100: emacs' ~editorconfig~ module (enabled in
[[file:../init.el][init.el]]) sets ~fill-column~ from it, and ~display-fill-column-indicator-column~
stays at its ~t~ default to track that; nvim's built-in editorconfig sets
~textwidth~, and ~+0~ tracks that. 100 = modern width that still survives a
side-by-side split.

the long-line TEXT is deliberately NOT painted: the loud option is
whitespace-mode's ~lines~ style (itself gated by ~whitespace-line-column~, nil =
~fill-column~), and the off-switch is just leaving ~lines~ out of the
whitespace-style, which this module does (a stale daemon on the default style
still shows the red tail until restarted). pinned by
~parity/long-line-hint-is-subtle-rule-not-aggressive-colour~.

** Completion sources (nvim-cmp parity map)

Which completion source is active, and in what order. Unlike nvim-cmp (one
merged menu), emacs' ~completion-at-point-functions~ is a CHAIN: the first
capf that matches wins and shadows the rest, so order and any explicit
merging matter. The cmp-* entries below detail each; this is the summary.

| nvim-cmp source | emacs capf                   | active in         | priority / notes                                             |
|-----------------+------------------------------+-------------------+--------------------------------------------------------------|
| path            | ~cape-file~                  | prog, text, conf  | first (depth -10); wins for ~./~ ~../~, descends into dirs    |
| lsp             | eglot capf                   | prog, lsp-attached | primary code source                                          |
| luasnip         | ~yasnippet-capf~             | modes with yas on | alongside lsp (~:editor snippets~)                           |
| buffer          | ~cape-dabbrev~               | prog, text, conf  | fallback (depth 20); in text, merged and ranked ABOVE spell  |
| spell           | ~ispell-completion-at-point~ | text only         | lowest; merged after buffer so its many words don't bury it  |
| lsp signature   | eglot + eldoc                | prog              | eldoc/hover, not a corfu menu source                         |

# short cells here, so this table survives outside emacs (no folded links) -
# unlike the "Where things go" table above; still nicer via the `gloworg' alias

In text-mode the buffer + spell sources are combined into one
~cape-capf-super~ (so both show at once, buffer first); everywhere else the
chain stands, with path highest and buffer as a low-priority fallback.

** DONE cmp-path (complete ~./~ and ~../~ file paths as you type)
nvim: the ~path~ cmp source is global
([[file:../../../../../nvim/lua/plugins/config/completion.lua][completion.lua]]), so paths complete in any buffer. emacs: doom's corfu
module already wires ~cape-file~, but only into prog-mode - extended to
the text/conf mode families in [[file:./config.el][config.el]] (org/markdown derive from

or ~../~ pops on its own; minibuffer paths were already fine via the

follow-up (2026-07-13): completing a DIRECTORY should keep offering its
contents (nvim cmp-path descends as you go deeper), but corfu stopped at the
inserted ~dir/~ - ~cape-file~ ships no ~:exit-function~, so nothing
re-triggered completion. added ~+neovim/cape-file-continue-in-dir-a~
([[file:./config.el][config.el]], a ~:filter-return~ advice) that injects an exit-function
re-invoking ~completion-at-point~ whenever the inserted candidate ends in
~/~; fires for RET (a dir picked from several) and TAB (a lone dir) alike and
recurses as you descend. this is also why ~+neovim/corfu-tab~'s lone-match
branch is now ~corfu-insert~ rather than ~corfu-complete~ (the exit-function
drives the directory continuation, so the simpler insert is enough). pinned
by ~parity/corfu-file-descends-into-directory~.

** DONE cmp-buffer (complete words from the buffer) (2026-07-13)
nvim's ~buffer~ cmp source (completion.lua, ~keyword_length = 4~) suggests
words already typed in the buffer - "apple" is there, type ~app~, it
completes. emacs had nothing: doom gates the equivalent (~cape-dabbrev~)
behind the corfu module's ~+dabbrev~ flag, which ~(corfu +icons)~
([[file:../../../init.el][init.el]]) doesn't set, so ~cape-dabbrev~ was absent from the capf list
(prog buffers had only ~cape-file~). wired ~cape-dabbrev~ in directly
([[file:./config.el][config.el]], ~+neovim/add-cape-dabbrev-h~ on the prog/text/conf hooks) at
depth 20, a low-priority fallback after lsp/file - the ordering nvim also
gives its ~buffer~ source. corfu-auto's 2-char prefix triggers it a touch
earlier than nvim's keyword_length 4, which is the responsive feel it's
wanted for. pinned by ~parity/buffer-word-completion~.

edge case (2026-07-13): nvim-cmp merges every source into one menu, but
emacs' ~completion-at-point-functions~ is a CHAIN - the first capf that
returns matches wins, even when both are ~:exclusive no~ (that only means
"fall through if I match nothing", not "merge"). so in a prose buffer the
spell source (~ispell-completion-at-point~, text-mode only) shadowed
~cape-dabbrev~: typing ~curr~ next to a ~current-base16~ showed only
dictionary words until the word was finished. merged the two with
~cape-capf-super~ ([[file:./config.el][config.el]],
~+neovim/merge-spell-and-buffer-completion-h~) so spelling and buffer words
appear together, added ahead (depth -1) of the standalone entries so it wins
and ~cape-file~ (depth -10) still takes real paths first. scoped to
text-mode (prog's flyspell only flags; conf has no spell capf to shadow).
prog-mode's lsp-vs-dabbrev shadowing is the same chain limitation but left
as-is (lsp dominates and an eglot super-capf is riskier). pinned by
~parity/prose-completion-merges-spell-and-buffer~.

ordering follow-up (2026-07-13): merged, the spell source's dozens of
dictionary words buried the one buffer word - "current-base16" sank to the
BOTTOM. source order alone can't fix it: the completion style (hotfuzz) sorts
by fuzzy score, and cape-super's ~:display-sort-function~ is ~identity~
(keeps that score order). ~corfu--sort-function~ consults
~corfu-sort-override-function~ FIRST (ahead of any display-sort-function), so
set it buffer-locally (text-mode) to ~+neovim/corfu-buffer-words-first~,
which partitions candidates that appear as a token in the buffer to the front
(one buffer scan, ~[[:alnum:]_-]+~ so hyphenated dabbrev words like
current-base16 count whole). spelling stays visible, just below the buffer
words. pinned by ~parity/corfu-buffer-words-first-sort~ (unit) and
~parity/prose-buffer-words-rank-first~ (live corfu popup).

** DONE ~TAB~ completes a lone candidate (2026-07-12)
nvim's ~TAB~ (nvim-cmp, [[file:../../../../../nvim/lua/plugins/config/completion.lua][completion.lua]]) makes a single match feel
like it just completes. doom binds corfu's ~TAB~ to ~corfu-next~
(config/default's ~cmds-tab~ menu-item), which - together with this
module's ~corfu-preselect 'first~ and ~corfu-cycle t~ - re-selects the only
candidate when just one matches: no visible change, looks like a bug (you'd
have to press ~RET~ to insert). fixed with ~+neovim/corfu-tab~ ([[file:./config.el][config.el]],
bound over ~corfu-map~'s ~TAB~/[[tab]]): cycle when ~corfu--total~ > 1,
otherwise ~corfu-complete~. chose ~corfu-complete~ over ~corfu-insert~ so a
lone matched DIRECTORY keeps completing into its contents, like nvim's path
completion. ~S-TAB~ (reverse cycle) left untouched. regression caught in
review: a custom wrapper name matches neither ~corfu-continue-commands~ nor
its built-in ~\`corfu-~ regex, so ~corfu--post-command~ quit the popup after
one press (cycled once, then closed/selected) - fixed by registering
~+neovim/corfu-tab~ in ~corfu-continue-commands~, as doom does for its own
~+corfu/*~ wrappers. all three now covered by ert (~parity/corfu-tab-*~ in

** DONE shared-daemon leakage across ~emacsclient -nw~ instances (2026-07-15)
$EDITOR is ~emacsclient -nw~ ([[file:../../my-custom/client-enhance/][client-enhance]]), so each terminal that opens a file
is a tty client frame on the one shared daemon - lightweight, but daemon
state is global. observed: terminal B's ~q~ surfaced terminal A's file, the
two frames' cursors moved together, and quitting one quit the other.

two causes, both leaning on doom's per-frame workspaces:
- FRAME side: doom's ~+workspaces-associate-frame-fn~ sends the first/lone
  client frame to the SHARED ~main~ (only later ones get a fresh ~#N~), so
  clients opened one-at-a-time all pile into ~main~ and share it.
  ~client-enhance/isolate-client-frame~ ([[file:../../my-custom/client-enhance/config.el][config.el]], installed as
  ~persp-emacsclient-init-frame-behaviour-override~) gives every tty client
  its own ~#N~; GUI clients (-c) keep doom's default.
- QUIT side: smart-quit ([[file:./smart-quit.el][smart-quit.el]] branch 4) used a GLOBAL
  ~doom-real-buffer-list~, so B's last-buffer ~q~ saw A's file and fell back
  to it. scoped the buffer/unsaved checks to the current workspace
  (~+workspace-buffer-list~), and added a branch - a tty client whose OWN
  workspace emptied finishes the client (returns to its terminal) instead of
  switching into another frame's workspace.

verified programmatically (~pty.fork~ execs real ~emacsclient -nw~ against a
throwaway daemon, drained so frame redraws don't block it): two clients land
in distinct ~#1~/~#2~ and their files stay put -
~(("main") ("#1" "a.txt") ("#2" "b.txt"))~. pinned by
~client-enhance/tty-client-frames-isolated-not-shared-main~ and
~smart-quit/emacsclient-frame-empty-own-workspace-finishes-client~; the whole
smart-quit matrix still passes.

caveat: the override installs at config load, so a "still leaks" is a stale
daemon - needs a full restart (not just ~doom/reload~) and fresh sessions.
